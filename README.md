# Локальный запуск через Docker Compose

1.  Клонируйте репозиторий:
    `git clone https://github.com/capfer27/Laravel-Blog-API.git`

2.  Выполните следующие команды:
    `make up`
    `make queue`

Примечание: Доплнительно можно запустить команду `make help` и посмотреть какие еще команды доступные.

3. Импортировать файл Laravel-Blog-API.postman_collection.json в Postman. Файл находится в корне проекта.

4. Протестировать

# API Endpoints - Prefix: /api/v1/blog

POST /posts – Create post (status = moderation)

GET /posts?status=published – List posts, optional filter

GET /posts/{id} – Show one post

DELETE /posts/{id} – Soft delete

# How Moderation Works

После создания публикации запускается задача в очереди с задержкой в ​​10 минут.
Задача случайным образом присваивает статус «опубликовано» или «отклонено», устанавливает дату модерации (modered_at) и дату публикации (published_at), если публикация опубликована.
Генерируется уведомление (запись в журнале - файл storage/logs/laravel.log), подтверждающее изменение.

# Почему именно Jobs, а не Scheduler?

Гарантия однократного выполнения — для каждого поста запускается отдельное отложенное задание; нет необходимости отслеживать состояние «уже модерировано».

Точное время — ровно 10 минут после создания, не зависит от интервала выполнения планировщика.

Простота — задания инкапсулируют всю логику без дополнительных запросов к базе данных или управления планировщиком.

Масштабируемость — асинхронная обработка не блокирует запрос API, и несколько рабочих процессов могут обрабатывать пиковые нагрузки.

Оба подхода могут работать, но **Jobs** здесь предпочтительнее, потому что:

Модерация должна происходить **ровно один раз** для каждого поста, **через 10 минут после создания**.
Планировщику пришлось бы постоянно запрашивать посты, которым ровно 10 минут, обрабатывать дубликаты и управлять флагом состояния (например, `moderation_job_dispatched`).

Отложенная задача гарантирует выполнение в нужное время без дополнительной обработки.

Jobs **отдельны** – ответ API не задерживается; основная/отложенная работа происходит в фоновом режиме.

Планировщики лучше подходят для **повторяющихся пакетных задач** (например, ежедневная очистка), а не для отдельных, чувствительных ко времени действий для каждой сущности.
